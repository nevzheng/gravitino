# API design references

Use the smallest relevant set. These are review inputs, not a mandate to adopt
every convention.

| Topic                                                                                                                   | Primary source                                                                         | Use it for                                                                                                                                                                                                                                                                                                                                                          |
| ----------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| HTTP resource and method semantics, status codes, conditional requests, content negotiation, and authentication headers | [RFC 9110 — HTTP Semantics](https://www.rfc-editor.org/rfc/rfc9110.html)               | Method/status/header correctness and safe/idempotent behavior. A `401` needs an applicable [`WWW-Authenticate` challenge](https://www.rfc-editor.org/rfc/rfc9110.html#section-15.5.2).                                                                                                                                                                              |
| Cache controls and validators                                                                                           | [RFC 9111 — HTTP Caching](https://www.rfc-editor.org/rfc/rfc9111.html)                 | `Cache-Control`, `ETag`, `Vary`, and conditional request review.                                                                                                                                                                                                                                                                                                    |
| Structured HTTP errors                                                                                                  | [RFC 9457 — Problem Details](https://www.rfc-editor.org/rfc/rfc9457.html)              | Error-envelope alternatives, problem types, and disclosure boundaries. It does not require every API to use `application/problem+json`.                                                                                                                                                                                                                             |
| OpenAPI contract structure                                                                                              | [OpenAPI Specification — latest published](https://spec.openapis.org/oas/latest.html)  | Paths, operations, media types, reusable components, security schemes, and `info.version`. Model authentication with Security Scheme Objects: an OpenAPI `Authorization` header parameter is ignored by the specification.                                                                                                                                          |
| JSON Schema behavior                                                                                                    | [JSON Schema Draft 2020-12](https://json-schema.org/draft/2020-12/json-schema-core)    | Requiredness, nullability, unions, formats, and unknown-property policy.                                                                                                                                                                                                                                                                                            |
| Resource-oriented API guidance                                                                                          | [Google AIP index](https://google.aip.dev/)                                            | Practical conventions. Start with [AIP-122 resource names](https://google.aip.dev/122), [AIP-131 standard methods](https://google.aip.dev/131), [AIP-158 pagination](https://google.aip.dev/158), [AIP-180 backward compatibility](https://google.aip.dev/180), [AIP-185 versioning](https://google.aip.dev/185), and [AIP-193 errors](https://google.aip.dev/193). |
| Semantic API release policy                                                                                             | [Semantic Versioning 2.0.0](https://semver.org/spec/v2.0.0.html)                       | Contract/version compatibility policy; distinguish it from route selection.                                                                                                                                                                                                                                                                                         |
| Bearer-token use                                                                                                        | [RFC 6750 — OAuth 2.0 Bearer Token Usage](https://www.rfc-editor.org/rfc/rfc6750.html) | `Authorization` headers and token exposure concerns.                                                                                                                                                                                                                                                                                                                |
| Rate-limit fields                                                                                                       | [RFC 9333 — RateLimit Fields for HTTP](https://www.rfc-editor.org/rfc/rfc9333.html)    | Consistent rate-limit response headers, if rate limiting exists.                                                                                                                                                                                                                                                                                                    |
| API threat model                                                                                                        | [OWASP API Security Top 10](https://owasp.org/API-Security/)                           | Authorization, object-level access, resource consumption, and sensitive-data review.                                                                                                                                                                                                                                                                                |
| Telemetry vocabulary                                                                                                    | [OpenTelemetry semantic conventions](https://opentelemetry.io/docs/specs/semconv/)     | Trace/correlation and HTTP observability vocabulary.                                                                                                                                                                                                                                                                                                                |

When a project has a published API policy, an established client ecosystem, or
a compatibility promise, treat that evidence as at least as important as a
general style guide. Cite the exact source used in the review.
